# ADR 0002: 개발 운영용 RAG 우선 구축

날짜: 2026-05-12  
상태: Accepted

## Context

StockRush는 MSA, Kafka, Saga, Outbox, CQRS, AI 기반 개발 운영 체계를 포함하는 프로젝트다.

범위가 넓기 때문에 설계 문서, ADR, API 명세, 테스트 전략, 코드 요약이 빠르게 늘어난다. 이 자료들이 흩어지면 AI Agent가 오래된 기준이나 잘못된 맥락을 기반으로 작업할 위험이 커진다.

프로젝트 내부 기능으로 고객지원 RAG를 넣는 선택지도 있지만, 현재 포트폴리오의 핵심은 커머스 핵심 흐름과 AI 기반 개발 운영 방식이다.

## Decision

서비스 내부 기능용 RAG보다 개발 운영용 RAG를 먼저 구축한다.

개발 운영용 RAG는 다음 역할을 맡는다.

- 프로젝트 규칙 검색
- agent-rules 기준 문서 참조
- ADR 검색
- 아키텍처 문서 검색
- API 명세 검색
- DB schema 요약 검색
- 코드 요약 검색
- 작업별 Context Pack 생성

## Alternatives

### 내부 기능용 RAG

예: 고객이 환불 정책이나 주문 상태를 질문하면 RAG로 답변한다.

장점:

- 서비스 기능으로 AI를 직접 보여줄 수 있다.
- 고객지원 도메인과 자연스럽게 연결할 수 있다.

단점:

- 커머스 핵심 아키텍처 구현보다 우선순위가 낮다.
- 프로젝트 초반에 넣으면 핵심 주문 흐름 구현을 밀어낼 수 있다.

### RAG 없이 문서만 수동 관리

장점:

- 초기 구현 비용이 낮다.
- 문서 구조만 잘 잡으면 단기적으로 충분하다.

단점:

- 문서가 늘어날수록 검색과 맥락 구성이 반복 비용이 된다.
- AI Agent 작업의 근거 추적이 약해진다.

## Consequences

긍정적 결과:

- AI Agent가 매 작업마다 필요한 규칙과 설계 맥락을 빠르게 가져올 수 있다.
- 작업 결과를 문서와 코드에 다시 반영하고 재색인하는 루프를 만들 수 있다.
- 포트폴리오에서 agentic development system을 구체적으로 설명할 수 있다.

비용:

- Phase 0에 도구 개발 시간이 추가된다.
- 검색 품질을 유지하려면 문서 구조와 metadata 기준을 관리해야 한다.

## Implementation Notes

- 초기 구현 위치는 `tools/dev-rag`로 한다.
- 초기 저장소는 SQLite를 우선 사용한다.
- 검색은 keyword search를 먼저 구현하고, vector search는 확장 지점으로 둔다.
- 출력은 Markdown Context Pack으로 한다.
- Context Pack에는 적용 규칙, 관련 설계, 관련 코드, 구현 주의점, 검증 명령을 포함한다.
